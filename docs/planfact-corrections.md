# Единый план приведения клона ПланФакта к каноническому поведению

## БЛОК A. Критические смысловые ошибки расчётов (P0)

### A1. [P0] ДДС строится по типу СТАТЬИ, а не по типу ОПЕРАЦИИ
- **Что не так:** `cashflow_report` берёт только операции `type in (income, outcome)` (+ move отдельно). Денежные операции по статьям `kind=asset/liability/capital` (выдача/возврат займа, взнос в капитал, покупка ОС за деньги) в ДДС не попадают.
- **Как правильно:** критерий попадания в ДДС — денежность операции (`type in income/outcome/move`, есть `op_date`, `status=committed`), НЕЗАВИСИМО от `kind` статьи. Тип/секция статьи (`activity` + `balance_section`) используется только для выбора группы строки (Операционная/Инвестиционная/Финансовая), а не для включения/исключения. Начисления (accrual), Отгрузки (shipment), Поставки (supply) — НЕ включать.
- **Где менять:** `backend/services/reports.py::cashflow_report`.

### A2. [P0] ДДС: денежный поток по видам деятельности неполный
- **Что не так:** активы/обязательства/капитал не формируют самостоятельных денежных строк.
- **Как правильно:** маппинг секций в потоки: Операционный = Доходы/Расходы + оборотные активы (3.1) + краткосрочные обязательства (4.1); Инвестиционный = внеоборотные активы (3.2); Финансовый = долгосрочные обязательства (4.2) + прочий капитал + вложения учредителей + дивиденды. Move — внутри потоков (через системные статьи, см. C5).
- **Где менять:** `cashflow_report`.

### A3. [P0] ДДС: остатки на начало/конец — точная формула
- **Что не так:** не гарантировано, что перемещения и `opening_balance` входят в остаток на начало.
- **Как правильно:** Остаток на начало = `Σ opening_balance счетов + (Σ income + Σ move-зачисления) − (Σ outcome + Σ move-списания)` по `op_date < period_start`, committed. Остаток на конец = начало + Общий поток периода.
- **Где менять:** `cashflow_report`.

### A4. [P0] ОПиУ: кассовый метод не включает начисления с `isOpuCalculation`
- **Что не так:** cash = только committed income/outcome по `op_date`. Канон: Cash = committed-платежи + начисления с `is_opu_calculation=true`.
- **Как правильно:** в method=cash учитывать `type in (income, outcome, accrual)`; для accrual — только при `is_opu_calculation == true` (явно; `false`/None = исключить). Shipment/supply из cash исключить.
- **Где менять:** `pnl_report (cash)`; модель `Operation.is_opu_calculation: bool` (nullable, без дефолта true).

### A5. [P0] ОПиУ: метод начисления не включает Отгрузки/Поставки
- **Что не так:** в accrual учитываются только income/outcome по `accrual_date` и `type=accrual`. Shipment/Supply живут только внутри сделок и в ОПиУ не попадают.
- **Как правильно:** в method=accrual включать `shipment` (доходная сторона) и `supply` (расходная) по `accrual_date` при `accrual_confirmed=true`, группируя по `kind` статьи части (income→Доходы, outcome→Расходы).
- **Где менять:** `pnl_report (accrual)`; enum `Operation.type += shipment, supply`.

### A6. [P0] Отчёты группируют по `kind` категории целиком, а не по типам частей `items[]`
- **Что не так:** `items[]` несут только `amount/category_id/project_id`. Одна операция может иметь части разных типов (Assets + Liabilities), часть может попадать в другой период по своей дате начисления.
- **Как правильно:** все отчёты (ОПиУ/ДДС/Баланс) разбивают операцию по `items[]` и берут тип из `category.kind` каждой части; агрегируют суммы по частям, а количество операций — по уникальному `operation_id`.
- **Где менять:** модель `OperationItem += calculation_date, value_in_user_currency` (+ опц. `deal_id`, `counterparty_id`); все три отчёта в `reports.py`.

### A7. [P0] Запрет fallback `calculation_date → op_date` и инвариант accrual-дат
- **Что не так:** риск подстановки `op_date`, когда нет даты начисления.
- **Как правильно:** в accrual НЕ подставлять `op_date`; нет даты начисления = часть не попадает в начислительный период. Для `type=accrual` принудительно `op_date == accrual_date` (валидация/автозаполнение).
- **Где менять:** `pnl_report`; валидация в `api`/модели операции.

### A8. [P0] Сделки: прибыль считается ручной формулой `amount − cost`
- **Что не так:** сделка сведена к ручным `amount/cost/received`; нет вычисляемых показателей и метода расчёта.
- **Как правильно:** ввести вычисляемую модель и `accounting_method = cash|calculation` (по умолчанию **calculation**):
  - `paid_value` = Σ committed Income (sale) / Outcome (purchase) по `op_date`;
  - `provided_value` = Σ подтверждённых отгрузок/поставок (`is_calculation_committed=true`) по `provided_date`;
  - `deal_value` (только purchase) = `max(Σ позиций, Σ выплат, Σ поставок)`;
  - calculation: `income = provided_value`, `outcome = Σ привязанных расходов+начислений по accrual_date (confirmed)`; cash: `income = paid_value по Income`, `outcome = Σ расходов по op_date (committed)`;
  - `profit = income − outcome` (**только для sale**; purchase — null);
  - `profitability = profit/income*100`, `null` при `income==0` (только sale).
- **Где менять:** `backend/services/deals-calc`; модель `Deal += accounting_method, closed`; `Shipment += provided_date, is_calculation_committed`, разделить sale→Shipment / purchase→Supply.

### A9. [P0] Сделки: задолженность не выводится, должна считаться из оплат и отгрузок
- **Что не так:** долг по сделке не считается вовсе.
- **Как правильно:** `shipped = Σ shipment.amount`, `paid = paid_value`; долг = `paid − shipped` с классификацией (sale: paid>shipped → неденежная кредиторка/аванс; paid<shipped → денежная дебиторка; purchase зеркально). `profit` продаж по факту отгрузки = `Σ shipment.amount − Σ shipment.cost`.
- **Где менять:** `deals-calc`.

### A10. [P0] Баланс: главное тождество и `is_balanced` отсутствуют
- **Что не так:** нет проверки сходимости на каждую дату.
- **Как правильно:** для каждого периода `is_balanced = (assets − (liabilities + capital)) == 0` (с допуском на округление). При расхождении — вернуть предупреждение, НЕ подгонять числа (не «затыкать» разницу нераспределённой прибылью).
- **Где менять:** `backend/services/reports.py::balance_report`.

### A11. [P0] Баланс: Капитал вынесен в top-level, а должен быть внутри Пассивов
- **Что не так:** `capital` — отдельная корневая секция наравне с активами/обязательствами.
- **Как правильно:** Пассивы = Обязательства + Капитал; Капитал — третья группа внутри `liabilities` (после кратко- и долгосрочных обязательств). Тождество: `Активы = Пассивы`.
- **Где менять:** `balance_report`; `Category.balance_section`.

### A12. [P0] Баланс: формула нераспределённой прибыли неполная (нет курсовой разницы)
- **Что не так:** `retained = прибыль − дивиденды`.
- **Как правильно:** `retained = income − outcome − dividends + course_diff`. Разбить на «Текущий год» (текущий результат + дивиденды + course_diff) и «Прошлый период» (накопленный результат прошлых периодов). `course_diff` = переоценка валютных остатков/обязательств к валюте отчёта на дату.
- **Где менять:** `balance_report`; расчёт/поле `course_diff`.

### A13. [P0] Дивиденды и Выручка определяются по подстроке/эвристике
- **Что не так:** дивиденды ищутся по подстроке «дивиденд» в названии `kind=capital`; «выручка» = все `kind=income`.
- **Как правильно:**
  - Дивиденды — явный признак статьи `Category.is_dividend` (системная статья 5.3), не по названию. Спец-правило: при `status=committed` у дивидендов требовать `accrual_confirmed=true`. Это единственная Capital-статья в расчёте нераспределённой.
  - Выручка по умолчанию = «Нераспределённый доход» (1.1) + «Выручка от продаж» (1.2); «Прочие доходы» (1.3), проценты, курсовые — исключать (только по явному флагу). Развести показатели «Доход» (весь Income) и «Выручка».
- **Где менять:** `pnl_report`, `balance_report`; модель `Category.is_dividend`, под-классификация income (`is_other_income` или подтип).

## БЛОК B. Методы расчёта и подтверждения (P0/P1)

### B1. [P0] Трёхзначные методы расчёта вместо бинарного cash/accrual
- **Что не так:** везде `method: cash/accrual`; `Cashflow ≠ Cash`; enum для контрагентов и проектов разные.
- **Как правильно:** раздельные enum:
  - Контрагенты: `Cashflow` (только committed, без начислений), `Cash` (committed + начисления с `is_opu_calculation=true`), `Calculation` (по `is_calculation_committed`).
  - Проекты: `Cashflow`, `IncomeExpensesByOperationDate`, `IncomeExpensesByCalculationDate`.
- **Где менять:** `contractors-calc`, `projects`-расчёты, `api`, `frontend`.

### B2. [P0] Три флага подтверждения операции вместо двух
- **Что не так:** есть `status=committed` + `accrual_confirmed` (последний используется для двух разных смыслов).
- **Как правильно:** три разных признака: `is_committed` (платёж), `is_opu_calculation` (начисление в ОПиУ-cash), `is_calculation_committed` (метод Calculation / дебиторка-кредиторка as-of). Развести текущий `accrual_confirmed` на `is_opu_calculation` и `is_calculation_committed`.
- **Где менять:** модель `Operation`/`OperationItem`; все отчёты.

### B3. [P0] Глобальный отсев операций без релевантной даты
- **Что не так:** фильтрация только по статусу.
- **Как правильно:** во ВСЕХ методах исключать операции без релевантной даты (нет `op_date` для cash-методов / нет `accrual_date` для accrual-методов), помимо проверки статуса.
- **Где менять:** все расчёты в `reports.py`, `contractors-calc`, `projects`.

### B4. [P1] As-of date — верхняя граница включительно
- **Что не так:** возможна граница `< dateEnd`.
- **Как правильно:** во всех финансовых endpoint-ах `date <= dateEnd` (за указанную дату входит). Особенно `balance_report` и `contractors-calc`.
- **Где менять:** все отчёты.

## БЛОК C. Модель данных: типы, статьи, счета (P1)

### C1. [P1] Недостающие типы операций
- **Что не так:** `Operation.type` = 4 значения.
- **Как правильно:** добавить `shipment`, `supply` как самостоятельные типы (неденежные: вне ДДС, вне cash-ОПиУ, вне остатков счетов, без движения по счёту). См. A5, D-список.
- **Где менять:** enum `Operation.type`, `api`, `deals-calc`.

### C2. [P1] Недостающие признаки статьи (Category)
- **Что не так:** нет ключевых атрибутов.
- **Как правильно — добавить в `Category`:**
  - `is_system` (🔒 — запрет ред/удаления/смены parent/kind);
  - `is_hidden` (🚫 BaseViewRestriction — скрыта из выбора в UI, есть в API);
  - `special_attributes` (массив/набор bool): `SelectOperationRestriction`/`SelectParentOperationCategoriesRestriction` (контейнеры — нельзя назначать операции), `SelectIncome/Outcome/AccrualOperationRestriction`; игнорировать `SelectAccrualOutcomeByIncome...`, `...ForCapital`, `SelectAccrualOutcomeOperationRestriction`;
  - `monetary` bool (денежная — на дату баланса; неденежная — на дату операции);
  - `payment_date_only` (= `SelectCalculationDateRestriction` — запрет/игнор `accrual_date`/`accrual_confirmed`);
  - `level`/контейнер (category / group_1 / group_2 / item) — 4 уровня; запрет операций на контейнеры;
  - `cost_type` (direct/indirect/none), `cost_behavior` (variable/fixed/none);
  - `is_dividend` (см. A13);
  - `is_depreciation`, `is_loan_interest` (для EBITDA/EBT, см. D2).
- **Где менять:** модель `Category`, валидация в `api`, `frontend pages` (статьи).

### C3. [P1] `activity` должен быть детерминирован веткой дерева
- **Что не так:** `activity` — свободное поле.
- **Как правильно:** валидировать/выводить из `balance_section`: current_asset/short_liability/доходы/расходы → operating; noncurrent_asset → investing; long_liability + весь capital → financing.
- **Где менять:** модель/валидация `Category`.

### C4. [P1] Денежная/неденежная логика в балансе
- **Что не так:** баланс единообразно использует accrual_confirmed/paid.
- **Как правильно:** денежные статьи (счета ДС, дебиторка, кредиторка, займы, фин.вложения) — as-of дата баланса; неденежные (запасы, ОС, НМА, доходы/расходы/капитал) — на дату операции.
- **Где менять:** `balance_report`.

### C5. [P1] Системные парные статьи для Move
- **Что не так:** Move — отдельная ветка без привязки к системным статьям.
- **Как правильно:** завести системные скрытые статьи «Отправленные платежи» (3.1.4.3, Assets) и «Полученные платежи» (4.1.2.3, Liabilities); Move проводить через них (для сходимости баланса). Модель Move: пара объектов с `bound_move_operation_id`; в UI сворачивать в одну строку суммой `0`; количество — по уникальному id связки.
- **Где менять:** модель `Operation` (`bound_move_operation_id`), системные статьи, `balance_report`, список операций.

### C6. [P1] Служебный (undistributed) счёт и реклассификация начислений
- **Что не так:** нет `Account.is_undistributed`; начисления сделаны через `debit/credit_category_id` без привязки к служебному счёту.
- **Как правильно:** добавить `Account.is_undistributed`; начисления привязывать к нему; операции на таком счёте не считать денежным движением (вне ДДС/остатков), даже если type похож на income/outcome.
- **Где менять:** модель `Account`, `Operation`, `cashflow_report`, расчёт остатков.

### C7. [P1] Статусы/архивность и ограничения записи
- **Что не так:** нет блокирующих правил.
- **Как правильно:** добавить `Project.closed`, `Project.archived` (отдельно от `active`), `Counterparty.archived`, `Account.archived` (+ архивность ЮЛ счёта), `Deal.closed`. Правила: closed-проект — запрет привязки И отвязки; archived-проект/контрагент/счёт — запрет новой привязки; closed-сделка — read-only (bind/unbind/modify запрещены). Валидацию выполнять ДО сборки payload с указанием блокирующей сущности.
- **Где менять:** модели, `api` (POST/PUT операций и сделок).

### C8. [P1] Ограничения привязки операций/отгрузок к сделкам
- **Что не так:** привязка неявная через `deal_id`, без проверки совместимости.
- **Как правильно:** к sale — Income/Shipment/расход; к purchase — Outcome/Supply. Для Shipment/Supply валидировать статью: разрешены доходы/расходы, активы (Запасы, Прочие оборотные, ОС, НМА), обязательства (Прочие краткосрочные, Платежи третьим лицам); запрещены Прочие внеоборотные (кроме НМА) и Долгосрочные обязательства.
- **Где менять:** `api` bind/unbind, `deals-calc`.

## БЛОК D. Полнота отчётов (P1)

### D1. [P1] ОПиУ: разрезы `report_gen_method = OperationCategory | Projects | Deals`
- **Как правильно:** добавить параметр; группировка по статьям / по проектам (группа «Без проекта» В КОНЦЕ, не исключать) / по сделкам (sale/purchase). Для каждого — Доходы/Расходы/Прибыль/Рентабельность. Опция «Остальные» (`with_other`).
- **Где менять:** `pnl_report`, `api`, `frontend`.

### D2. [P1] ОПиУ: профит-метрики и классификация расходов
- **Как правильно:** флаги `is_gross_profit/operating_profit/ebitda/ebit/ebt` (по умолчанию НЕ считать). Формулы: Валовая = Выручка − Прямые; Операционная = Валовая − Косвенные; EBITDA = Операционная + Амортизация; EBIT = EBITDA − Амортизация; EBT = EBIT − Проценты по кредитам. Прочие расходы (амортизация 2.7–2.10, проценты, налог на прибыль) — после операционной прибыли. Требуются признаки статьи из C2 (`cost_type`, `is_depreciation`, `is_loan_interest`).
- **Где менять:** `pnl_report`, `Category`.

### D3. [P1] ОПиУ: рентабельность — прочерк при убытке
- **Как правильно:** при `outcome > income` (прибыль < 0) рентабельность = `null`/прочерк, не отрицательный процент.
- **Где менять:** `pnl_report`.

### D4. [P1] ОПиУ: режим План + Факт
- **Как правильно:** факт = committed; план = те же типы без требования committed; отдельная колонка «План». Уточнить дату попадания начислений в кассовый ОПиУ (особый нюанс из OPERATIONS).
- **Где менять:** `pnl_report`.

### D5. [P1] ДДС: разрезы по проектам и сделкам
- **Как правильно:**
  - Projects: поток = committed income + move-зачисления − committed outcome − move-списания; исключить accrual/shipment/supply; остатки по счетам НЕ выводить; остатки на начало/конец — по проекту; «Без проекта» в конце.
  - Deals: операционный + инвестиционный потоки, БЕЗ финансового.
  - Убрать любой параметр метода cash/accrual у ДДС-endpoint.
- **Где менять:** `cashflow_report`, `api`, `frontend`.

### D6. [P1] Баланс: структура секций и денежно/неденежное деление
- **Как правильно:** стандартизованные узлы: оборотные — «Денежные средства» (с разбивкой нал/безнал/карты физлиц/электронные по `Account.kind`), «Запасы», «Переводы в пути», «Дебиторская задолженность» (денежная/неденежная), «Другие оборотные»; внеоборотные — «Основные средства»/«Другие»; краткосрочные — «Кредиторская задолженность» (денежная/неденежная)/«Другие»; долгосрочные — «Кредиты»/«Другие»; капитал — «Вложения учредителей»/«Нераспределённая прибыль»/«Другие». «Переводы в пути» = move со списанием подтверждённым, зачислением — нет. Правила вывода счетов: пустой счёт без операций и нач.остатка — скрыть; счёт с операциями и итогом 0 — показать; архивные — строкой «Счета в архиве» последней. Поддержать `standard_period` и серию колонок `values_by_period` (каждая со своим `is_balanced`, `course_diff`).
- **Где менять:** `balance_report`; модель Category (подкатегории), `frontend`.

### D7. [P1] Дебиторка/кредиторка: денежная/неденежная + просрочка
- **Что не так:** считается «по разрыву дат», смешаны денежная/неденежная, нет просрочки, нет полей.
- **Как правильно:** 4 комбинации с явным `debt_kind = monetary|non_monetary`: Income оплата подтв./начисл. нет → неденежная кредиторка (аванс получен); Income оплата нет/начисл. подтв. → денежная дебиторка; Outcome оплата подтв./начисл. нет → неденежная дебиторка (аванс выдан); Outcome оплата нет/начисл. подтв. → денежная кредиторка. По контрагенту вернуть `all_credit/past_credit/all_debit/past_debit`, `cash_debt/non_cash_debt`, `income/outcome/profit`, `operations_count`. Долг as-of считать методом `Calculation`. Просрочка = неподтверждённая операция с датой < as-of (cash: `op_date<as_of & not committed`; accrual: `accrual_date<as_of & not is_calculation_committed`). Нужна дата срока оплаты для просрочки.
- **Где менять:** `contractors-calc`, модель операции/счёта (срок оплаты), `balance_report`.

### D8. [P1] Финансовые агрегаты по проектам
- **Как правильно:** добавить измерение `project_id` в бюджет; расчёт `projects/calculations`: `income_plan/outcome_plan` (из бюджета, НЕ из суммы операций) + `income_fact/outcome_fact` (выбранным методом) одним ответом; `invoice_balance` (дебиторка) и `invoice_supply_balance` (кредиторка) по проекту; статус `New/InProgress/Done`; флаг `closed`. Не суммировать строки как ОПиУ компании. Фильтр «в работе»: исключать проекты с `income_fact==0 && outcome_fact==0`. Cash-policy: UI-aligned (включать нераспределённое, default) / strict, помечать вариант в ответе.
- **Где менять:** новый расчёт `projects-calc`, модель `Budget` (+`project_id`), `Project` (статус/closed), `api`.

## БЛОК E. Бюджеты (P1)

### E1. [P1] `Budget.budget_method` (bdr/bdds) на уровне сущности
- **Как правильно:** обязательный атрибут; БДР и БДДС независимы (нельзя копировать структуру/настройки между типами). Перенести разделение БДР/БДДС с фронта в бэкенд (фильтр, набор статей, метод факта).
- **Где менять:** модель `Budget`, `api`, `frontend`.

### E2. [P1] Метод БДР (cash/accrual) как атрибут бюджета; факт тем же методом
- **Что не так:** факт по бюджету всегда committed.
- **Как правильно:** для БДР добавить `accrual_basis`/метод. Факт считать ТЕМ ЖЕ методом: БДР-accrual → `accrual_date`/`accrual_confirmed`; БДР-cash и БДДС → committed по `op_date` (+ перемещения для БДДС, начисления исключить).
- **Где менять:** план-факт расчёт бюджета.

### E3. [P1] БДДС: включать asset/liability/capital, поток по `activity`
- **Как правильно:** в БДДС включать денежные статьи всех типов (не только income/outcome); поток наследуется из `category.activity`; перемещения внутри потоков; preflight: если в факте есть Investment/Finance, а в плане только Operating — ошибка маппинга.
- **Где менять:** БДДС-построение, валидация.

### E4. [P1] БДР: доходы и расходы в одном бюджете
- **Как правильно:** хранить доход/расход единым набором BudgetItem, различать по `category.kind`; не разносить в отдельные сущности. Плановые строки писать на конец месяца (если генерируется конкретная дата).
- **Где менять:** модель/логика бюджетов.

Метод (cash/accrual) — параметр блоков Прибыль/Топ-клиенты/Проекты; если не указан — уточнять, не подставлять молча.

## БЛОК H. Мелочи и поведенческие правила (P2)

- **H1. [P2]** Остаток счёта — только вычисляемый (нач + приходы − расходы ± перемещения); хранимый `remainder` нигде не отдавать как источник истины.
- **H2. [P2]** Количество операций — по уникальному `operation_id`, не по числу `items[]`/строк; суммы — по частям.
- **H3. [P2]** Move в списке операций — отображаемая сумма `0`; raw списание/зачисление — только технический оборот внутри ДДС.
- **H4. [P2]** Shipment/Supply — входят в список операций и в count, но НЕ в денежные итоги (поступления/выплаты/поток = только Income/Outcome committed).
- **H5. [P2]** Все агрегации — по `base_amount` (valueInUserCurrency), не по сумме в валюте счёта.
- **H6. [P2]** Пустой результат (`isSuccess:true`, нули/пустой items) = «данных за период нет», не ошибка; не расширять период автоматически — спросить пользователя.
- **H7. [P2]** Убрать из кода клона несуществующую ссылку «§6.8» (нет основания в каноне).
- **H8. [P2]** Маппинг терминов для интеграции с реальным API: `op_date↔operationDate`, `accrual_date↔calculationDate`, `accrual_confirmed↔isCalculationCommitted`, `status==committed↔isCommitted`, `is_opu_calculation↔isOpuCalculation`, `base_amount↔valueInUserCurrency`.
